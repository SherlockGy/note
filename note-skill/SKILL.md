# note-skill：笔记查询、合并与清理

## 目录结构

```
~/.notes/
├── *.md                                 # 正常笔记（文件名=标题）
├── .config/categories                   # 分类配置（每行一个）
├── .config/tags                         # 标签配置（每行一个，去重）
├── merged/                              # 被合并的旧笔记
└── deleted/                             # 被清理的笔记
```

**重要：默认查询只搜索 `~/.notes/` 根目录下的 `.md` 文件，不搜索 `merged/` 和 `deleted/` 子文件夹。** `merged/` 中的内容已在合并后的新笔记中，无需搜索。用户明确要求查看已删除的笔记时，可搜索 `deleted/`。

## 笔记格式

每条笔记是 Markdown 文件，带 YAML frontmatter：

```yaml
---
category: 技术
tags: [docker, mysql]
created: "2026-03-12 周四 14:30"
updated: "2026-03-12 周四 14:30"
merged_from:                      # 可选，合并笔记时记录来源
  - Docker端口映射.md
  - Docker网络配置.md
---
```

文件名就是标题、标题就是文件名。frontmatter 中没有 title 字段。frontmatter 字段顺序：`category` → `tags` → `created` → `updated` → 可选字段。

## 一、查询

用户意图示例："查一下我关于 docker 的笔记"、"我之前记过一个 MySQL 的解决方案"、"最近的笔记有哪些"

### 执行策略：两步定位法

所有查询遵循同一流程，不按意图分支。基于三种跨平台原子操作：**LIST**（列目录）、**SEARCH**（跨文件搜文本）、**READ**（读单个文件）。

#### 第 1 步：扫描（1-2 次操作，得到候选列表）

**操作 A — LIST**：列出 `~/.notes/*.md` 所有文件名和修改时间（1 次目录操作）。文件名就是标题，这一步同时获得全部标题和日期。

**操作 B — SEARCH**（如有关键词）：在 `~/.notes/*.md` 中搜索关键词（1 次内容搜索操作）。一次搜索同时覆盖 frontmatter（tags、category）和正文。

A 和 B **并行执行**。如果用户只想看「最近的笔记」，只需操作 A 按日期排序，无需操作 B。

将 A 和 B 的结果合并为候选列表，按命中优先级排序：
1. 标题命中（文件名包含关键词）
2. 标签命中（frontmatter tags 包含关键词）
3. 正文命中（正文包含关键词）

同一笔记命中多层时，只出现在最高优先级层。

#### 第 2 步：读取（只读候选文件）

根据候选数量决定展示方式：
- **≤ 3 条**：全部读取，展示内容摘要
- **4-10 条**：列出标题 + 标签概览，让用户选择要看哪条
- **> 10 条**：提示用户缩小关键词范围
- **0 条**：告知未找到，建议换个关键词或用 `note search` 命令

**完成后告知用户**：扫描了多少条笔记，命中了几条。

#### 语义扩展（AI 的独特优势）

用户描述模糊时，在操作 B 中扩展同义词搜索。例如：
- "数据库" → 同时搜索 `mysql|postgres|sql|db|数据库`
- "部署" → 同时搜索 `deploy|docker|k8s|ci/cd|部署`

这是 AI 相比 CLI `note search` 的核心优势：理解语义、跨笔记关联分析、总结归纳。

#### 禁止事项

- ✗ 逐个读取所有笔记文件来搜索（用 LIST + SEARCH 代替）
- ✗ 跳过第 1 步直接读文件（先扫描再定位）
- ✗ 搜索 `merged/` 和 `deleted/` 目录（除非用户明确要求）

## 二、合并

用户意图示例："帮我把这几条 Docker 相关的笔记合并一下"、"整理一下我的 React 笔记"

### 合并规则

合并规则同 CLI `note merge` 命令（见命令行接口设计 §6），核心要点：
- **不得增加原本不存在的内容**：合并后的笔记中，每一句话都必须能在原笔记中找到出处。只允许调整讲述方式（重新组织段落顺序、消除重复、统一格式），不允许补充新知识、新解释、新示例。（此规则是 AI 合并的核心约束，CLI 简单拼接天然满足）
- 创建新 .md 文件，旧笔记移入 `merged/`
- frontmatter 中 `merged_from` 标注来源，正文中 `<!-- from: 原文件名.md -->` 标注段落来源
- 标签合并去重，分类投票取最常见
- 更新 `.config/tags`

**AI 合并 vs CLI 合并的区别：** AI 可以理解内容语义，做更高质量的整合（消除重复、重新组织结构）。CLI 只做简单按序拼接。

### 合并完成后告知用户

输出格式：
```
✅ 合并完成

新笔记：Docker网络与端口配置完全指南.md
分类：技术
标签：docker, mysql, 网络, 端口
来源（已移入 merged/）：
  - Docker端口映射.md
  - Docker网络配置.md
  - Docker连宿主机MySQL.md

合并说明：将 3 条 Docker 相关笔记整合为一份，消除了重复的网络配置说明，按主题重新组织了段落顺序。未添加任何原笔记中不存在的内容。
```

## 三、清理

用户意图示例："这条笔记没用了，清理掉"、"删掉那几条过期的备忘"

### 清理规则

1. **不是真删除**：将笔记移到 `~/.notes/deleted/` 文件夹，而非永久删除
2. **确保目录存在**：`mkdir -p ~/.notes/deleted`
3. **操作前确认**：列出即将清理的笔记标题，请用户确认
4. **保留文件原样**：移动时不修改文件内容

### 清理完成后告知用户

输出格式：
```
🗑️ 清理完成

已移入 deleted/：
  - 过期的临时备忘.md
  - 旧版部署流程.md

共清理 2 条笔记。如需恢复，可在 ~/.notes/deleted/ 中找到原文件。
```

## 通用规范

### 每次操作完成后，必须告知用户工作内容

不管是查询、合并还是清理，完成后都要输出一段简洁的工作总结，包含：
- **做了什么**：查询了什么 / 合并了哪些 / 清理了哪些
- **结果**：找到几条 / 新建了什么文件 / 移动了什么文件
- **影响**：哪些文件被移动了位置（如移入 merged/ 或 deleted/）

### 搜索范围

| 场景 | 搜索范围 |
|------|----------|
| 默认查询 | `~/.notes/*.md`（仅根目录） |
| 用户要求查看已删除的 | `~/.notes/deleted/*.md` |
| 用户要求搜索全部 | `~/.notes/*.md` + `~/.notes/deleted/*.md` |

注意：`merged/` 不纳入任何查询范围，其内容已在合并后的新笔记中，搜索合并后的笔记即可。
